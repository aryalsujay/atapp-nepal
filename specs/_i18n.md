# Internationalization (i18n) — Workflow & Conventions

This app supports English (`en`) and Nepali (`ne`). All user-visible text flows through a JSON-file-based translation system so translators can update strings without touching code.

## 1. Where strings live

```
src/translations/
├── en.json    # English — the source language
└── ne.json    # Nepali — must mirror en.json's key structure
```

Add `xx.json` for a new language and register it in `src/i18n/index.ts`.

## 2. How it's wired

`src/i18n/index.ts` loads both files at app boot via `react-i18next`. Every screen calls `useTranslation()` and uses `t('namespace.key')`:

```tsx
import { useTranslation } from 'react-i18next';

function MyScreen() {
  const { t } = useTranslation();
  return <Text>{t('login.title')}</Text>;
}
```

The current language is held in `src/store/settingsStore.ts` and `app/_layout.tsx` re-mounts the navigation tree on language change so every screen picks up the new strings.

## 3. Key naming conventions

| Convention | Why |
|---|---|
| `namespace.action_name` | one namespace per screen or feature; `snake_case` for keys |
| `login.email_label`, `login.cta_teacher` | scoped to screen / role / action |
| Avoid sentence-as-key (`t("Welcome back")`) | hard to track changes; breaks if EN text shifts |
| Avoid nested keys deeper than 2 levels | `login.errors.invalid_admin` is fine; deeper is annoying to maintain |
| Prefer descriptive over short | `login.error_invalid_teacher` > `login.err4` |

Standard namespaces (current):
- `common` — generic verbs ("Save", "Cancel", "OK")
- `login` — login screen
- `home`, `courses`, `applications`, `profile`, `notifications` — screen-scoped
- `teacher`, `admin`, `server` — role-scoped extras

## 4. How to edit a translation

**Translator's workflow (no engineer needed):**

1. Open `src/translations/en.json` or `ne.json`
2. Find the namespace (e.g. `"login": { ... }`)
3. Change the value (not the key). Example:
   ```diff
   - "forgot": "Forgot password?"
   + "forgot": "Reset password"
   ```
4. Commit the change. The string updates everywhere on the next build / hot reload.

**Rules:**
- Never change a **key** — keys are the contract between code and translation. Only change values.
- If you need a new line / paragraph break in NE/EN, use `\n` inside the JSON string.
- Emojis are allowed inline (`"🌿 Server"`).
- Both `en.json` and `ne.json` must have **the same set of keys**. Missing keys fall back to English, which surfaces as untranslated EN text in the NE build.

## 5. How to add a new key

1. Add to `en.json` first (source language):
   ```json
   "login": {
     "remember_me": "Remember me on this device"
   }
   ```
2. Add the same key to `ne.json` with the translated value:
   ```json
   "login": {
     "remember_me": "यो उपकरणमा सम्झनुहोस्"
   }
   ```
3. Use it in code: `t('login.remember_me')`
4. Document the new string in the screen's spec under §6 (Strings & i18n).

## 6. How to add a new language

1. Copy `en.json` to `xx.json` (e.g. `hi.json` for Hindi)
2. Translate each value (keep keys identical)
3. Register in `src/i18n/index.ts`:
   ```ts
   import hi from '../translations/hi.json';
   ...
   resources: {
     en: { translation: en },
     ne: { translation: ne },
     hi: { translation: hi },
   }
   ```
4. Update `settingsStore.ts` to allow `'hi'` as a language option
5. Add the language to whichever switcher UI exists (currently only settings screen)
6. If the language uses a non-Latin script, load its font in `_layout.tsx` (see Noto Sans Devanagari pattern)

## 7. Fonts for non-Latin scripts

Latin scripts use **Plus Jakarta Sans**. Devanagari (Nepali) uses **Noto Sans Devanagari**.

Both are loaded in `app/_layout.tsx` via `@expo-google-fonts/*` packages and exposed via `FontFamily.*` tokens in `src/theme/typography.ts`. When rendering Devanagari text, use `fontFamily: FontFamily.devanagari` (or `devanagariBold` etc.).

For a new script (e.g., Arabic), install the matching `@expo-google-fonts/noto-sans-arabic` package and add to `useFonts()`.

## 8. Interpolation & pluralization

`react-i18next` supports both, but the prototype doesn't currently use them. Patterns:

**Interpolation** — `"greeting": "Namaste, {{name}} 🙏"` → `t('home.greeting', { name: 'Ananda' })`

**Pluralization** — `"course_count_one": "1 course"`, `"course_count_other": "{{count}} courses"` → `t('home.course_count', { count: 47 })` automatically picks the right plural form.

Use these sparingly — too many runtime substitutions make translation files hard to scan.

## 9. Testing translations

There's no automated coverage check yet. Manual test:

1. Run the app in EN, navigate through every screen
2. Switch to NE via settings, repeat
3. Watch for: untranslated EN appearing in NE build, text overflow on narrow screens, line-height issues with Devanagari
4. Special test: long Nepali compound words can overflow buttons — verify CTA text fits

> **Future improvement:** add a lint script that diffs the key sets of `en.json` and `ne.json` and fails CI if they diverge.

## 10. Export / import for translators

The whole translation file is portable. To hand off to a translator:

1. Email or share `ne.json` (or whichever file)
2. They edit values, send back
3. You replace the file → commit → ship

No tools, no Lokalise, no CMS needed for Nepal-scale.

If you eventually want a translator-friendly format, consider:
- Convert to/from `.po`/`.pot` (gettext)
- Or use a service like Crowdin / Lokalise that imports/exports JSON

Don't add this until there's a real translator pain point.
